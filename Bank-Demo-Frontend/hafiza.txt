## 4. KEYCLOAK OIDC GİRİŞ AKIŞI (Token Exchange)

### Akış Şeması (Login Flow)
```
1. Kullanıcı /login sayfasında "Giriş Yap" butonuna tıklar
                ↓
2. Frontend, Keycloak'ın Authorization Endpoint'ine yönlendirir:
   https://auth.bank.local/realms/bank-realm/protocol/openid-connect/auth
   ?client_id=bank-auth-client
   &response_type=code
   &redirect_uri=https://app.bank.local/api/auth/callback
                ↓
3. Keycloak giriş ekranını gösterir (TC No + Şifre)
                ↓
4. Başarılı girişte Keycloak, callback URL'ye "code" parametresiyle yönlendirir:
   https://app.bank.local/api/auth/callback?code=XXXX
                ↓
5. /api/auth/callback/route.ts (SERVER TARAFINDA):
   - code ile Keycloak'ın Token Endpoint'ine POST yapar
   - KEYCLOAK_INTERNAL_URL kullanılır (Docker iç ağı: http://keycloak:8080)
   - grant_type=authorization_code ile access_token ve id_token alır
   - Cookie'lere set eder: token (access_token), id_token
   - /auth-sync sayfasına yönlendirir
                ↓
6. /auth-sync/page.tsx (CLIENT TARAFINDA):
   - Cookie'den token'ı alır
   - customerService.getProfile() ile Backend'den profil bilgisi çeker
   - JWT'yi Base64 decode ederek Keycloak claim'lerini okur (name, email)
   - Backend verisi + Keycloak verisi harmanlanarak final profil oluşturulur
   - Zustand store'a (useAuthStore) login() ile kaydedilir
   - Role göre ilgili dashboard'a yönlendirilir
```

### Kritik Notlar
- Token Exchange: Server Route (/api/auth/callback) kullanılarak client_secret
  tarayıcıya ASLA ifşa edilmez
- KEYCLOAK_INTERNAL_URL: Token exchange sırasında Docker iç ağından geçer
  (http://keycloak:8080), HTTPS sertifika sorunları bypass edilir
- id_token: Keycloak logout işleminde id_token_hint parametresi için saklanır

## 5. GÜVENLİK VE KALKANLAR (proxy.ts / Middleware)
Middleware iki aşamalı bir duvar örer:

### Aşama 1: Token Ölüm Kontrolü (Expiration Check)
- Cookie'den token alınır, Base64 ile JWT payload decode edilir
- exp claim'i kontrol edilir (payload.exp < Date.now()/1000)
- Süre dolmuşsa: Keycloak Logout URL'sine yönlendirilir
  - post_logout_redirect_uri: NEXT_PUBLIC_FRONTEND_URL
  - id_token_hint: id_token cookie'sinden alınır
  - Cookie'ler (token, id_token) silinir

### Aşama 2: Yetki Duvarları (Role-Based Access Control)
| Koşul                                | Sonuç                              |
|--------------------------------------|------------------------------------|
| Token yok + korumalı rota            | → /login'e yönlendir               |
| Token var + /login veya /register    | → Role göre dashboard'a fırlat    |
| /admin/* + ADMIN rolü yok            | → /login'e yönlendir               |
| /company/* + CORPORATE_MANAGER yok   | → /user/dashboard'a yönlendir     |
| /user/* + ADMIN rolü var             | → /admin/dashboard'a yönlendir    |
| /user/* + CORPORATE_MANAGER rolü var | → /company/dashboard'a yönlendir  |

### Matcher (Hangi rotalarda çalışır?)
```
/login, /register, /admin/:path*, /user/:path*, /dashboard/:path*, /company/:path*
```

## 7. TEMEL İŞ KURALLARI
1. Vitrin Kurları: Ana sayfada currency-service'ten /api/v1/currencies/rates
   endpoint'inden kur bilgileri çekilir (Gateway Sivil Bölge, Auth gerektirmez)
2. Axios İstek Yönetimi: Tüm servis dosyaları (services/*.ts) token'ı
   cookie'den alıp Authorization: Bearer <token> header'ıyla Gateway'e gönderir
3. Profil Harmanlama: auth-sync sayfasında Backend profili + Keycloak token
   claim'leri birleştirilerek Zustand'a kaydedilir (isim/email Keycloak'tan,
   rol/statü Backend'den)
